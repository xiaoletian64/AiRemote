# Remote Bridge 产品设计文档（PRD）

> 把小米蓝牙电视遥控器变成 PC 的无线键盘+麦克风桥接器
>
> Version: v0.1 (草稿) · Date: 2026-04-28 · Owner: yuezhang

---

## 0. 一句话定位

**一台插在 PC 上的小盒子，让小米电视遥控器能操控电脑。**

- 即插即用：插 USB → PC 看到键盘+麦克风
- 自动接管：遥控器配对一次后，开箱就用
- 中间透明：用户感觉是在用一个被破解的好用遥控器

---

## 1. 用户画像

### 主用户：长期工作于 PC 的开发者/媒体用户

- 已经手头有小米遥控器（电视盒子带的，闲置心疼）
- 想在 PC 上有个远端的"啊先按个空格/方向键"的设备
- 还想要近端的语音输入（替代昂贵的语音麦克风）
- 不愿意折腾 Linux/驱动配置
- 能接受「插上 USB 啥也不用配」就最爽

### 次用户：HTPC / 客厅 PC 家庭媒体中心

- 小米遥控器比无线键鼠更适合远端操控
- 语音查电影/搜索很自然
- 不在乎是小米还是其他品牌，能接就行

### 反面用户（不做为他们设计）

- 想用通用蓝牙手柄当 PC 输入设备的（这是另一个产品）
- 想做整套 BLE 中间人攻击的安全研究者（这是另一个开源方向）

---

## 2. 关键体验原则（PK · Principle & Key）

1. **零驱即用**：插 Windows/Mac/Linux 不弹驱动安装窗
2. **状态可见**：用户任何时候都能从灯光读出"盒子现在啥状态"
3. **失败可恢复**：配对错设备？想换遥控器？一键清掉重来
4. **配置可选**：80% 用户即插即走，20% 用户能改键映射也能记住设置
5. **远端友好**：5 米外能识别状态（灯光足够明显）

### 4.2 LED 信号规范（颜色 + 闪烁模式语义）

| 状态 | 颜色 | 模式 | 用户感受 |
|---|---|---|---|
| 未配对/待配对 | 白 | 慢呼吸 1Hz | "我空着，来配一个" |
| 扫描中 | 蓝 | 慢呼吸 1Hz | "在找遥控器" |
| 配对中 | 蓝 | 快闪 4Hz | "正在握手" |
| 已连接(Active) | 绿 | 常亮 | "正常工作" |
| 按键按下脉冲 | 绿 | 闪一下(50ms) | "传输中"（可选演示模式，默认关） |
| 重连中 | 绿 | 慢闪 1Hz | "在重连" |
| 配置模式 | 紫 | 常亮 | "进配置模式了" |
| 错误 | 红 | 快闪 4Hz | "出错了" |
| OTA 升级 | 橙 | 常亮 → 呼吸 | "在升级别拔" |

### 4.3 BOOT 键交互规范

| 操作 | 时长 | 触发效果 |
|---|---|---|
| 短按 (<200ms) | — | 在 Active/RECONNECT_WAIT 间切换显示模式（演示用，默认仅记日志） |
| 双击 | — | 立即重新发起一次扫描（覆盖了，找不到也不强制） |
| 长按 3s | 3s | 进配置模式（紫色常亮） |
| 长按 5s | 5s | 清配对（白灯亮一下确认 → 自动进扫描） |
| 长按 10s | 10s | 进 OTA 固件升级模式 |

## 5. 三段式用例旅程

### 5.1 首次使用（unboxing → first keypress）

```
[1] 拆开盒子
    └─ 用户开 ESP32-S3 + USB-C 线，没有别的说明纸（仅盒底一短 QR）

[2] 插 PC
    └─ LED 红色一闪（USB 枚举）→ 白色慢呼吸
    └─ PC 弹出键盘+音频设备（系统原生提示，无需驱动）

[3] 把遥控器按住 "Home + OK" 5 秒进入配对模式
    └─ 遥控器 LED 快闪
    └─ 在 30s 内，盒子 LED 自动从白 → 蓝慢闪 → 蓝快闪 → 绿色常亮
    └─ 用户感觉到"配上了"
    
[4] 按遥控器方向键
    └─ PC 上有响应（焦点移动、Tab、Enter 等等）
    └─ 视觉反馈完成闭环

[5] 按遥控器语音键说话
    └─ PC 上识别为麦克风输入设备
    └─ 用户在任意应用中选中此麦克风（首次需要手动选一次）
```

**首配即零配置是这个产品 ≥80% 的奄奄一息录取线。**

### 5.2 日常使用（已 Bond）

```
[1] 把盒子一直插在 PC 上（或插在显示器 USB Hub）
    └─ LED 绿色常亮

[2] 长时间不用遥控器
    └─ 遥控器进睡眠，盒子维持绿色常亮 30 分钟
    └─ 30 分钟无操作 → LED 调暗 50%（节能模式，不黑）

[3] 拿起遥控器按任意键
    └─ 遥控器从睡眠唤醒 → 自动广播
    └─ 盒子在 500ms 内识别 → 进入 Active（绿色恒亮）
    └─ 按键到达 PC

[4] 遥控器在我 PC 上启动中的应用里用
    └─ 默认：方向键=方向键，OK=回车，Back=Esc，Vol±=音量键
    └─ Home = 显示桌面 (Win+D / Cmd+F3 默认)
    └─ Voice 键=语音流送去 USB UAC 麦克风
```

### 5.3 配置（需要改默认时的进阶用户）

```
[1] 长按 BOOT 键 3 秒
    └─ LED 紫色常亮
    └─ PC 收到一个额外的 USB CDC ECM 虚拟网卡（或 USB 串口）

[2] PC 浏览器自动弹出 http://bridge.local
    └─ 如果没自动弹，用户手动打开 http://192.168.42.1 (SoftAP 时)

[3] Web 配置页内容见第 7 节

[4] 改完点"保存"
    └─ 配置写入 NVS 持久化
    └─ LED 紫色快闪 2 次确认
    └─ 自动返回 Active 模式
```

## 6. 状态机与事件流的同步

设备 FSM 和现有事件队列的映射：

| 设备状态 | 触发事件 | 来源 | 转发到 USB | LED |
|---|---|---|---|---|
| IDLE_NO_BOND → SCANNING | 60s 定时器 / BOOT 双击 | ble_central | — | 白→蓝呼吸 |
| SCANNING → PAIRING | ESP_GATTC_CONNECT_EVT | ble_central → app_event | — | 蓝→蓝快闪 |
| PAIRING → ACTIVE | SMP 完成 + Notify 订阅 | ble_central → app_event | — | 蓝闪→绿常亮 |
| ACTIVE → RECONNECT_WAIT | ESP_GATTC_DISCONNECT_EVT | app_event | — | 绿常亮→绿慢闪 |
| ACTIVE → ACTIVE | ESP_GATTC_NOTIFY_EVT (key) | app_event → usb_hid.send_key | ✓ | 绿 50ms 闪 |
| ACTIVE → ACTIVE | ATVV audio frame | app_event → usb_uac.send | ✓ | — |
| * → CONFIG_MODE | BOOT 长 3s | main.c (button poll) | — | 紫常亮 |

### 7.5 Tab④ 语音

- 语音输入设备名（默认 "Remote Bridge Mic"）
- 采样率/编码：16kHz / ADPCM（自动跟随遥控器）
- 静音检测（VAD）：
  - ○ 全程上传
  - ○ 自动检测说话段上传（默认）
- 推送模式：
  - ○ 按住语音键说话（默认，和遥控器原生一致）
  - ○ 推一下开始 / 再推一下结束
- 音量增益：[-12dB ~ +12dB]（默认 0dB）

### 7.6 Tab⑤ 系统

- LED 亮度：[0~100] 默认 60%
- LED 演示模式：[关/开脉冲]（Active 状态显示按键脉冲/不显示）
- USB 描述符：
  - 制造商字符串（默认 "Yue Bridge"）
  - 产品字符串（默认 "Remote Bridge"）
  - 序列号
- 蓝牙发射功率（仅高级）：[-12 ~ +9 dBm]
- 自动睡眠（无操作 N 分钟后切回 RECONNECT_WAIT）：[5/15/30/60/Off]
- 固件更新：
  - 当前版本: 0.1.0
  - [选择固件文件] [开始 OTA]
- 重启、恢复出厂

### 7.7 配置中立存储

- 配置存 NVS 命名空间 `bridge_cfg`
- 按键映射表：每条 = 4B（usage 16-bit + mode 8 + target 8）
- 12 键 → 48B，远低于 NVS 限制
- 出厂默认：可按 BOOT 长 5s 同时通过 WebUI "恢复出厂" 重置

## 8. PC 端的"无感知"部分（重点设计）

### 8.1 USB 复合设备呈现给 PC 的样子

```
USB Composite Device
├─ HID Keyboard         ——  遥控器方向键/OK/Back/Home/Menu
├─ HID Consumer Control ——  Vol± / Power (映射到 sleep)
└─ UAC Audio Input       ——  TV remote 麦克风
```

**视觉效果**：
- Windows 设备管理器 → 键盘类多出 "Remote Bridge"
- Windows 声音控制面板 → 录制设备多出 "Remote Bridge Mic"
- macOS → 系统报告 → USB 多一项，声音设置多一个输入源
- Linux → /dev/input/eventN + /dev/snd/pcmC1D0c

### 8.2 让用户"零选择"的小策略

- **键盘部分**：默认全场生效，用户不用做任何选择，当前焦点在哪个窗口，方向键就发给谁
- **麦克风部分**：
  - 启动时 box 主动发一次 "ping" 音到 UAC（2 秒后停止）
  - 让 OS 看到此设备有信号 → 不强制用户切换
  - 在 WebUI 里提示用户："如果你想让某一应用使用此麦克风，需要在应用选一次"

### 8.3 不做（避免过度设计）

- ❌ 不做桌面端配置 App（WebUI 够用，省去维护成本）
- ❌ 不做云账号/远程配对
- ❌ 不做热键全局拦截
- ❌ 不做翻译/ASR（语音流交给 PC 应用层处理，比如 whisper.cpp、Speech-to-Text）

## 9. 错误状态与可恢复性

| 问题 | 表现 | 用户视角 | 自动恢复 | 手动恢复 |
|---|---|---|---|---|
| 上电时 BLE 初始化失败 | LED 红色快闪 | "哎坏了吗" | 重启 1 次 | BOOT 长 5s 重置 |
| 已配对的遥控器失联 | 绿色慢闪 | "啊遥控器没用了吧" | 60s 自动重扫 | BOOT 双击立即重扫 |
| 配对中 passkey 错 | 蓝色变红一闪再回白 | "没配上，重试一下遥控器按键" | 自退回 | 遥控器按键 |
| USB 枚举失败 | LED 换插后熄灭 | "是 USB 没插好" | — | 拔插 |
| ATVV 语音流卡死 | 绿色变橙 2s | "语音按着没反应" | 重订阅 | 重启遥控器 |
| 配置写入 NVS 失败 | 紫色闪一下 | "改没存住" | — | 重试 |
| OTA 失败 | 橙色变红 | "升级挂了先别拔" | 自动回滚 | 全手动重烧 |

## 10. 隐私与安全

| 项 | 设计 |
|---|---|
| 配对密钥 | Secure Connections (BLE 4.2+) + Bond，避免 Just Works 的弱点 |
| Bond 存储 | 只在 NVS 中，不外泄 |
| 语音流 | 不在 ESP32 端保存/转发到云；直接走 USB 到 PC |
| WebUI 访问 | 仅 USB CDC ECM / 本地 SoftAP，不暴露公网 |
| WebUI 鉴权 | 无密码（本地访问即可），无远程能力 |
| OTA 包 | 默认从用户本地上传，不自动拉云 |

## 11. 对比同类产品定位（竞品）

| 产品 | 做什么 | 我们做的差异化 |
|---|---|---|
| Stream Deck | 程序化可编程键盘 | 我们是远端/无线/语音，依赖已有遥控器复用 |
| Key Mapper / Karabiner | PC 端改键映射 | 我们做的是桥层改映射，不依赖 PC 端 Agent |
| 巴掌大的 BLE HID Host 桥 (Raspberry Pi Zero 做) | 同思路 | 我们便宜/低功耗/单芯片 ESP32-S3 ≈15元 |
| 一般 BLE 键盘重映射器 | 同方向近 | 我们特定遥控器 + 语音流 |
| PC 端 "Bluetooth HID Host" 系统自带 | Windows 仅支持带 Microsoft BLE HID Profile 的设备 | 我们桥掉这个限制，给任意 BLE HID 设备开放 |

## 15. 总结：合理性的几个关键决策

| 决策 | 为什么"合理" |
|---|---|
| 不做桌面对端 App | 节省维护成本 90%，体验损失 5%。ROI 高 |
| 默认映射走"原生含义" | 上手不需要学习，看到的遥控器就是它在电视上那个角色 |
| LED + 单键作为全部物理交互 | BOM 成本 ≈ 0；5 项状态足够；参考 AirPods 耳机的状态灯思路 |
| 配对放权给遥控器原厂模式（Home+OK） | 不强制重新发明配对方式；复用用户的肌肉记忆 |
| 用 Bluedroid 不用 NimBLE | 实测兼容更稳，代价是 flash/RAM 多用一些 |
| 语音流不走 ASR | PC 端已有更强 ASR，我们做透传最薄 |
| WebUI 通过 USB CDC ECM | 不需要软 AP 切换网络，不需要 App，也不暴露网络安全面 |
| 不做云、不做账号 | 这不是一个 SaaS，是一个工具 |

## 附：第 4 节状态机与现有 main/app_fsm.c 对照

**现有 FSM**：仅消费按键事件，转发到 USB
**PRD 要求的 FSM**：

- POWER_ON → USB_INIT → IDLE_NO_BOND → SCANNING → PAIRING → ACTIVE → RECONNECT_WAIT
- 任意状态 → CONFIG_MODE (BOOT 长 3s)
- 任意状态 → IDLE_NO_BOND (BOOT 长 5s 清 bond)
- 任意状态 → ERROR (硬件/初始化失败)

**建议下一步：升级 `app_fsm.c` 加入状态机本身** —— 这是把产品逻辑落到代码的第一刀。
